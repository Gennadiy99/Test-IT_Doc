Добавление тега для нового commit

Для первого входа в сохранение данных на Git end Git-Hub нужно внести команды:

  1. git init
  2. git add .
  3. git commit -m "first commit" .
  4. git branch -M main
  5. git remote add origin "ссылка на репозиторий Git-Hub"
  6. git push -u origin main

Для последующих сохранений нужно только команы:

  1. git add .
  2. git commit -m "new commit"
  3. git push

В терминале мы можем прописать команду "git status --short" в место команды которая выведет полное описание втатуса "git status"

Если мы в Граф.ред. нажмем на "+" это то же самое что в терминале команда "git add ." Тоесть изменения будут добавленны в "стэйтж" - После чего при команде "git status --short" буква "М" будет подсвечиваться зеленой.

-- Далее нужно комитить и пушить что бы изменения попадали в репозиторий Git-Hub

Команда git log --oneline --graph --decorate --all - выдаст отобранную инормацию по камитам(без лишних данных)

Есть список команд для терминала, которые дают нам более точную информацию, такую как выаодит Git-Graph

git branch -M main - Может переименовывать ветки, НО может и СОЗДАВАТЬ.(расмотрим это позже.)

Мы можем открыть файл Git config которая находится на ПК в User, сделаем это через VS code. В этом файле можно создавать "alias - Алиас" - сокращенные команды. 

      [alias]
      s = status --short
      l = log --oneline --graph --decorate --all
      g = log --graph --abbrev-commit --decorate --all --format=format:'%C(bold blue)%h%C(reset) - %C(bold cyan)  %aD%C(dim white) - %an%C(reset) %C(bold green)(%ar)%C(reset)%C(bold yellow)%d%C(reset)%n %C(white)%s%C  (reset)'
      br = branch
      co = checkout

После этого мы можем в терминали писать сокращенную команду "git s" - это равнозначно "git status"

Можно сохранять сокращенную команду в таком уточнени как вам удобно, допустим: 

        br = branch -M main 
        Теперь команда br будет переименовывать ветку в "main"

Так же можно отменить изменения в ком.стр. написав:

Таким способом мы отменяем сохранения в файле (html/css ...)

Если мы сохранили изменение и добавили их в "промежуточные изменения".

Выполнили это сохранение через:

  • Через командную строку "git add ."
  • Через граф.инт "+" - "Индексировать изменения"
Отменить можем:
  • в Граф.интерф. нажатием на " - " минус. На надписи "Промежуточное значение"
  • Меню 3 точки - "Изменения" - "Отменить Промежуточные изменения"
  • в командной строке "git reset ." или "git reset inde.xhtml" (если изменения были только в нем)

Если сохранили изменения в "commit"

Сохранение в commit делается:

  • терминал командой git commit -m "new commit"
  • Написания названия комита в поле "Сообщение" + кнопка "Фиксация"в Граф.интр.

Отмена commita и изменения в файле делается:

  • ком.строке "git reset -- hard HEAD^1" (жесткий сброс 1 комит)
При этом удаляется комит и само изменение в html файле.

Если нам нужно просто удалить комит и оставить данные в файле для корректировки, то нужно сделать "мягкий сброс" делается єто в ком.стр. "git reset -- soft HEAD^1"

Если в последствии не захотите сохранять оставленные данные то пишем в ком.стр.

  • "git reset ." - Удаляем из Промежуточной зоны
  • "git checkout -- ." - Удаляем из сохраненных

Так же можно сделать мягкий сброс через меню 3 точки - "Зафиксировать" - "Отменить последнюю фиксацию"

Так же выбрав в "Git Graph" нужный комит + при помощи меню выбираем "checkout" что бы переключиться на работу с этой веткой.

4.12 Слияние веток Git

Ветки в Git можно использовать для комфортной работы одного или нескольких разработчиков над проектом. При помощи веток вы можете создавать ответвления от главного проекта и реализовать лишь нужный для вас функционал, а потом необходимо выполнить слияние веток и разрешить конфликты слияния.

Ветка отдельная копия проекта по состоянию на определённый коммит. В репозитории может быть множество отдельных веток. Почему полезно работать в отдельных ветках? Работа в ветках полезна при командной разработке, когда разные разработчики разрабатывают разную функциональность по одному и тому же проекту. Работа в отдельных ветках позволяет разным разработчикам работать в независимых копиях проекта и вносить изменения по своим разработкам, при этом не мешать друг другу своими изменениям.

В этом уроке мы рассмотрим:

  • Процесс слияния веток при помощи git merge
  • Конфликты и их разрешение
  • Слияние перемоткой Fast-forward

При ответвлении и изменениях в новой ветке, которая идет из основной, и код в ней нас устраивает, мы можем сделать слияние Новой ветки с Основной веткой. Для этого нам нужно перейти в основную ветку и сделать в ниее слияние Новой ветки После нужно написать команду: "git merge (new_branch)" Произойдет слияние Новой ветки (new_branch) в Основную ветку.

Такое слияние можно сделать и через "Git Graph" наж.пр.кл.мыш.-"Слить ветку в текущую"

Так же єто можно сделать через: меню 3 точки выбрав "Ветка"-"ОБьеденить Ветвь"

Рассмотрим Слияние Веток, когда обе ветки были изменены. (конликты-слияния)

При слиянии 2х измененных веток используется тот же принцип, и та же команда:"git merge new_branch". Нужно находится на той Ветке в которую будет сливаться вторая ветка.

В редакторе появится сообщение в случаии конфликта Веток и укажет в файле html места конфликта. Нужно в ручную удалить не нужный код в html и в граф.интерф. нажать галочку для фиксации коммита который будет уже заполнен сообщением (которое можно изменить)

Так же слияние веток можно сделать серез меню - точки - "Ветвь" - "Обьеденить Ветвь" Fix_4

Можно выполнить слияние веток через Git Graph, став на ветку которая будет Основной и в которую будет сливаться новая ветка.
